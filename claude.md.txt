# claude.md — Harness Engineering Playbook
이 파일은 “대화”가 아니라 “레포의 파일”을 기준으로 개발이 굴러가게 만드는 작업 하네스다.

---

## 0) 프로젝트 한 줄 요약 (WHY)
전 세계 사용자가 **한국어/일본어를 기초부터(또는 동시에)** 학습하고, 필요하면 **사전 검색 + AI 번역/간단 설명(일일 제한)**까지 할 수 있는 웹/앱 플랫폼을 만든다.

- 핵심 차별점: KR/JP 동시 학습 + 비교 학습 카드 + (선택) AI 설명

## 2) 🔒 Harness 핵심 추가 구성 (필수급/강추)
> 아래 구성은 **컨텍스트 흔들림 방지**를 위한 “고정 장치”다.  
> 4개 핵심 문서( PRD/계획/규칙2종 )만으로도 돌아가긴 하지만, 아래가 있으면 **세션 재개 비용이 급감**한다.

### 2-1) `progress/PROGRESS.md` (필수급)
**역할**
- 현재 작업 상태 / 다음 한 걸음 / 블로커를 한 파일에 고정
- 매 세션 시작 시 여기부터 읽고 재개

**없으면**
- 세션 끊길 때마다 “어디까지 했지?”가 반복되고 재개 비용이 커진다.

**형식 예시(아주 짧게)**
- 지금 테스크: `T-0001 Auth`
- 다음 액션: `DB 스키마 확정 → API 엔드포인트 작성`
- 블로커: `없음` (또는 이슈 링크/로그 경로)

### 2-2) `docs/tasks/` 폴더 (강추)
**역할**
- 테스크를 “말”이 아니라 **파일**로 관리 (T-0001, T-0002 …)
- 우선순위/범위/검증 기준이 문서로 고정되어 흔들리지 않음

**없으면**
- 에이전트가 테스크를 매번 새로 해석해서 **우선순위/범위가 흔들릴 확률이 급증**한다.

### 2-3) `progress/CHANGELOG.md` / `progress/FAILURES.md` (선택이지만 효과 큼)
- `CHANGELOG.md` : **결정이 바뀐 이유** 기록(나중에 흔들릴 때 복구 가능)
- `FAILURES.md` : 막힌 원인/해결 기록(같은 함정 2번 방지)

## 4) MVP 범위(오늘 우리가 “반드시” 만드는 것)
MVP 목표(우선순위 고정):

1) 회원가입/로그인  
2) 학습 모드 선택: KR / JP / BOTH  
3) 단계별 커리큘럼 + 학습 카드/퀴즈 + 진도 저장  
4) 백과사전 검색(영어 입력 → KR/JP 뜻 + 예문)  
5) 포인트 적립 + 기본 공유  
6) AI 기능 1종(텍스트): “내 언어 → KR/JP 번역 + 간단 설명” + **일일 제한**

Post-MVP: STT, 구독/결제, 이미지 변환 등 (MVP 완료 후)

---

## 5) 작업 방식(항상 동일한 루프)
### 5-1) 세션 시작 루틴(주니어도 그대로 따라하기)
1) `claude.md` 읽기
2) **GitHub URL 확인**: `0-1)` 섹션에 주소가 비어 있다면 사용자에게 요청하고 기록한다.
3) **File bootstrap** 수행(없으면 생성)
4) `progress/PROGRESS.md` 읽기 (현재 테스크/다음 액션/블로커 확인)
5) `docs/tasks/`에서 “지금 테스크 파일” 열기  
   - 없다면 `task_creation_rules.md` 기준으로 먼저 만든다.

### 5-2) 기본 실행 루프(반드시 지키기)
1) **Understand**: 테스크 문서를 읽고 5줄로 요약
2) **Plan**: 3~7개 마이크로 스텝으로 쪼개기 + 롤백 포인트 명시
3) **Execute**: 스텝마다 “작은 diff → 실행/테스트 → 결과 기록”
4) **Verify**: Acceptance Criteria를 체크박스로 검증
5) **Update**: PROGRESS/CHANGELOG/FAILURES 갱신
6) **Push**: 각 테스크 완료 시(Status -> DONE) 반드시 `git push`를 수행한다.
7) **Done**: 산출물 정리(문서/테스크 상태 변경)

> 목표는 “한 방에 크게”가 아니라 “작게, 검증하며, 되돌리기 쉽게”.

---

## 6) 테스크 운영 규칙(생성/실행 표준)
### 6-1) 테스크는 이렇게 만든다 (Creation Harness)
- 모든 테스크는 `docs/tasks/`에 `T-XXXX-<slug>.md`로 저장
- 테스크는 “작게, 검증 가능하게” (1개 사용자 가치 단위)
- MVP 우선순위: Auth/Settings → Learning Loop → Dictionary → Points/Share → AI(일일 제한) → Post-MVP

**테스크 템플릿(필수 필드)**
- Goal(사용자 가치)
- Scope(포함/제외)
- Deliverables(코드 + 검증 방법은 필수)
- Acceptance Criteria(검증 가능한 문장)
- Verification Plan(자동/수동 시나리오)

### 6-2) 테스크는 이렇게 실행한다 (Execution Harness)
절대 규칙:
- 작은 diff
- 매 스텝 검증
- 시크릿/개인정보 로그/커밋 금지
- “추측”이 필요하면 먼저 실험/재현/로그 확인

막히면(15분 이상):
- 재현 최소화 → 로그 첨부 → 롤백 → 가설 3개 + 실험 1개로 전환
- 그리고 `progress/FAILURES.md`에 남긴다

---

## 7) Definition of Done (DoD) — “완료”의 정의
테스크는 아래를 모두 만족해야 DONE:
- [ ] PRD/구현계획의 해당 요구를 충족
- [ ] 검증(테스트 or 구체적 수동 시나리오) 수행
- [ ] 회귀 위험이 있으면 최소한의 회귀 체크 추가
- [ ] `progress/PROGRESS.md`에 “무엇이 끝났고 다음은 무엇인지” 기록
- [ ] 관련 문서 업데이트(API/DECISIONS 등)
- [ ] 변경이 의사결정 수준이면 `progress/CHANGELOG.md`에 이유 기록
- [ ] **GitHub Push**: 모든 변경사항을 원격 저장소에 푸쉬 (`git push`)

## 8) AI 기능은 별도 하네스(비용/보안/일일 제한 때문에)
AI 관련 구현은 반드시 다음 순서:

1) `packages/ai-core/`에 Provider 인터페이스 정의
2) Mock Provider 구현(테스트 가능)
3) Gemini Provider 구현
4) Claude Provider 구현
5) `ai_usage`로 사용량 집계 + 일일 제한(서버 사이드)
6) 캐싱(동일 입력 재사용)
7) 비용/한도/실패 대응을 문서화

프롬프트는 코드에 박지 말고:
- `packages/ai-core/prompts/<feature>.txt`에 저장
- 입력/출력 스키마를 명확히 (가능하면 JSON)

## 11) 🎨 Design System — 2025 트렌드 (30년차 디자이너 기준)
> 이 섹션은 모든 UI 구현의 기준이다. 반드시 따라야 한다.

### 11-1) 철학
- **Cultural Fusion Minimalism**: 한국의 절제미(여백) + 일본의 정밀함(디테일)을 현대 UI에 융합
- **Dark-First**: 기본 테마는 다크. 라이트는 옵션
- **Motion as Language**: 애니메이션은 "보여주기용"이 아니라 상태 전환/피드백을 전달하는 수단
- **Thumb-first Mobile**: 모든 핵심 인터랙션은 엄지 한 손으로 가능해야 함

### 11-2) 컬러 팔레트
```
-- 배경 (Dark-first)
--bg-base:       #0D0D1A   /* 딥 인디고 블랙 */
--bg-surface:    #13131F   /* 카드/패널 배경 */
--bg-elevated:   #1C1C2E   /* 모달/드롭다운 */

-- 유리 효과 (Glassmorphism 2.0)
--glass-bg:      rgba(255,255,255,0.04)
--glass-border:  rgba(255,255,255,0.08)
--glass-blur:    backdrop-filter: blur(20px) saturate(180%)

-- 브랜드 (KR/JP 이중 정체성)
--brand-kr:      #FF4D6D   /* 한국: 코랄 레드 (활기, 열정) */
--brand-jp:      #C084FC   /* 일본: 사쿠라 퍼플 (우아함, 전통) */
--brand-both:    #6366F1   /* 동시학습: 인디고 (지식, 깊이) */

-- 포인트 / 게임화
--accent-gold:   #F59E0B   /* 포인트, 스트릭, 달성 */
--accent-green:  #10B981   /* 정답, 완료, 성공 */
--accent-red:    #EF4444   /* 오답, 경고 */

-- 텍스트
--text-primary:  #F0F0FF
--text-secondary:#8B8BA8
--text-muted:    #4A4A6A
```

### 11-3) 타이포그래피
```
-- 영문: Inter Variable (가변 굵기)
-- 한국어: Noto Sans KR
-- 일본어: Noto Sans JP
-- 학습 카드 표시 폰트: 크기 2배, weight 700, letter-spacing tight

font-sizes:
  display: 48~64px (랜딩 히어로)
  h1: 32px
  h2: 24px
  body: 16px
  caption: 13px
  card-word: 40px (학습 카드 메인 단어)
```

### 11-4) 핵심 컴포넌트 패턴

**Bento Grid (대시보드 레이아웃)**
- 모듈형 카드 그리드. 크기가 다른 카드들이 맞물리는 레이아웃
- CSS Grid: `grid-template-columns: repeat(12, 1fr)`

**Glassmorphism Card**
```css
background: rgba(255,255,255,0.04);
border: 1px solid rgba(255,255,255,0.08);
backdrop-filter: blur(20px) saturate(180%);
border-radius: 20px;
```

**학습 카드 (Flip Card)**
- 앞면: 영어 단어
- 뒷면: 한국어 + 일본어 나란히 (비교 학습)
- 전환: 3D CSS flip (perspective 1000px)

**모드 스위치 (KR / BOTH / JP)**
- 3-way pill toggle, 선택된 모드에 따라 accent color 변경
- KR → --brand-kr, BOTH → --brand-both, JP → --brand-jp

**Progress Bar**
- 얇은(4px) 그라데이션 바. brand color 기반
- 애니메이션: width transition 0.6s ease-out

### 11-5) 마이크로 인터랙션 규칙
- 버튼 hover: scale(1.02) + brightness(1.1), duration 150ms
- 카드 hover: translateY(-4px) + shadow 강화, duration 200ms
- 정답: ✅ 초록 flash + scale(1.05) → 정상, duration 400ms
- 오답: ❌ 빨간 shake(좌우 ±8px x3), duration 400ms
- 페이지 전환: fade + translateY(8px → 0), duration 200ms

### 11-6) 반응형 브레이크포인트
```
mobile:  < 640px  (Expo + 웹 모바일)
tablet:  640~1024px
desktop: > 1024px
```

### 11-7) 접근성 (WCAG AA)
- 모든 텍스트 대비율 4.5:1 이상
- 포커스 링 표시 (outline: 2px solid --brand-both)
- aria-label 필수 (아이콘 전용 버튼)

## 12) 🧪 TDD 기반 성능 최적화 — MVP 완료 후 적용

> **적용 시점**: MVP의 모든 기능(§4)이 DONE 상태가 된 이후에만 이 섹션을 실행한다.
> 기능 개발 중에는 테스트 코드보다 "작동하는 코드"가 우선이다.
> 단, 테스트하기 어려운 구조가 보이면 즉시 `progress/FAILURES.md`에 기록해둔다.

---

### 12-1) 원칙: TDD 사이클 (Red → Green → Refactor)

```
1. Red    : 실패하는 테스트를 먼저 작성한다
            "이 API는 200ms 안에 응답해야 한다"
2. Green  : 테스트를 통과하는 최소한의 코드를 작성한다
3. Refactor: 테스트가 통과된 상태에서 코드를 개선한다
            (성능 최적화, 가독성, 중복 제거)
```

> "테스트 없는 최적화는 추측이다. 테스트 있는 최적화만이 사실이다."

---

### 12-2) 테스트 레이어 구성

#### Layer 1 — 단위 테스트 (Unit Test)
- **대상**: 순수 함수, 서비스 메서드, 유틸리티
- **도구**: Jest (NestJS/백엔드), Vitest (Next.js/웹)
- **위치**: 각 파일 옆에 `*.spec.ts` 또는 `__tests__/` 폴더
- **커버리지 목표**: 핵심 비즈니스 로직 80% 이상

```ts
// 예시: auth.service.spec.ts
it('should hash password on signup', async () => {
  const result = await authService.signup({ email: 'a@b.com', password: '1234' });
  expect(result.user.passwordHash).toBeUndefined(); // 응답에 해시 노출 금지
});
```

#### Layer 2 — 통합 테스트 (Integration Test)
- **대상**: API 엔드포인트 전체 흐름 (Controller → Service → DB)
- **도구**: Jest + Supertest (NestJS), 테스트용 Postgres (docker-compose.test.yml)
- **위치**: `apps/api/test/`

```ts
// 예시: POST /auth/signup → DB에 유저 생성 확인
it('POST /auth/signup returns 201 and tokens', async () => {
  const res = await request(app.getHttpServer())
    .post('/auth/signup')
    .send({ email: 'test@test.com', password: 'pass1234' });
  expect(res.status).toBe(201);
  expect(res.body).toHaveProperty('accessToken');
});
```

#### Layer 3 — E2E 테스트 (End-to-End)
- **대상**: 실제 사용자 시나리오 전체 흐름
- **도구**: Playwright (웹 브라우저 자동화)
- **위치**: `apps/web/e2e/`
- **핵심 시나리오 (반드시 포함)**:
  1. 회원가입 → 로그인 → 대시보드 진입
  2. 학습 모드 변경(KR→JP→BOTH) → DB 반영 확인
  3. 레슨 시작 → 카드 플립 → 레슨 완료 → 포인트 적립

### 12-3) 성능 목표 수치 (Performance Budget)

> 아래 수치를 테스트 코드로 자동 검증한다. 수치를 넘으면 배포 차단.

#### 백엔드 API 응답 시간
| 엔드포인트 | 목표 | 한계치 |
|------------|------|--------|
| POST /auth/login | < 300ms | 500ms |

  - Next.js Image 최적화 (next/image 사용)

### 12-5) 테스트 자동화 파이프라인 (CI 연동)

```yaml
# .github/workflows/test.yml 에 추가할 내용 (요약)
on: [push, pull_request]
jobs:
  test:
    steps:
      - run: npm test              # 단위 테스트
      - run: npm run test:e2e      # 통합 테스트
      - run: npx lighthouse-ci     # 성능 예산 검증
```

- PR 머지 전 모든 테스트 통과 필수
- Lighthouse 점수 90 미만이면 PR 차단
- 테스트 실패 원인은 반드시 `progress/FAILURES.md`에 기록
